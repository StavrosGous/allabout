===========================================================
  AllAbout — Immersive 3D Knowledge Platform — Project Plan
===========================================================

An interactive 3D encyclopedia where users explore knowledge by navigating
and zooming through spatial scenes (e.g., science lab → microscope → cells
→ molecules → atoms).

Stack: React Three Fiber (frontend) + FastAPI on Uvicorn (backend) + MongoDB
Content sourced from Wikipedia/Wikidata + user contributions.
Anonymous browsing; login only for editing.

-----------------------------------------------------------
ARCHITECTURE OVERVIEW
-----------------------------------------------------------

Frontend:  React + React Three Fiber + drei + zustand
Backend:   FastAPI running on Uvicorn (Python ASGI server)
Database:  MongoDB (document store — naturally fits flexible schemas)
Storage:   S3-compatible (MinIO for dev, S3/Cloudflare R2 for prod)
3D Format: GLB with meshopt compression + KTX2 textures
Cache:     Redis (Wikipedia/Wikidata response cache)
Dev Env:   Docker Compose (MongoDB, MinIO, Redis, backend, frontend)

PHASE 2: DATA MODEL & API
-----------------------------------------------------------

6. MongoDB collections (Beanie ODM documents):

   users (collection)
   ├── _id: ObjectId (auto)
   ├── username: str (unique index)
   ├── email: str (unique index)
   ├── hashed_password: str
   ├── role: str (viewer | editor | admin)
   └── created_at: datetime

   knowledge_nodes (collection)
   ├── _id: ObjectId (auto)
   ├── slug: str (unique index)
   ├── title: str
   ├── summary: str
   ├── full_content: str (Markdown)
   ├── wikidata_id: str (optional)
   ├── wikipedia_url: str (optional)
   ├── node_type: str (concept | object | process | person | place)
   ├── properties: dict (flexible key-value: mass, atomic_number, etc.)
   ├── tags: list[str]
   ├── parent_id: ObjectId (optional, ref → knowledge_nodes)
   ├── relations: list[{node_id: ObjectId, type: str}]  ← embedded edges
   ├── created_by: ObjectId (ref → users)
   └── created_at, updated_at: datetime

   Note: relations are embedded in the document for fast reads.
   For complex graph traversals, use MongoDB $graphLookup aggregation.

   scenes (collection)
   ├── _id: ObjectId (auto)
   ├── slug: str (unique index)
   ├── title: str
   ├── description: str
   ├── parent_scene_id: ObjectId (optional, ref → scenes)
   ├── environment: dict (hdri, ambient_light, fog settings)
   ├── camera_defaults: dict (position, target, fov, near, far)
   ├── zoom_depth: int (0 = top-level)
   └── objects: list[SceneObject]  ← embedded sub-documents:
       ├── id: str (UUID string for frontend reference)
       ├── knowledge_node_id: ObjectId (ref → knowledge_nodes)
       ├── asset_id: ObjectId (ref → assets, optional)
       ├── transform: dict ({position, rotation, scale})
       ├── interaction_type: str (zoom_into | popup_info | link_to_scene | none)
       ├── zoom_target_scene_id: ObjectId (optional, ref → scenes)
       ├── label: str
       ├── highlight_color: str (optional)
       └── lod_levels: list[{distance: float, asset_id: ObjectId}]

   Note: scene_objects are embedded inside scenes since they are always
   fetched together. This avoids a separate collection + join.

   assets (collection)
   ├── _id: ObjectId (auto)
   ├── filename: str
   ├── storage_url: str (S3/CDN URL)
   ├── format: str (glb | gltf | ktx2 | hdr)
   ├── file_size_bytes: int
   ├── poly_count: int (optional)
   ├── thumbnail_url: str
   ├── license: str
   ├── status: str (pending | approved | rejected)
   ├── uploaded_by: ObjectId (ref → users)
   └── created_at: datetime

   Indexes:
   - knowledge_nodes: text index on title + summary (full-text search)
   - knowledge_nodes: index on tags, wikidata_id, slug
   - scenes: index on slug, parent_scene_id
   - assets: index on status, uploaded_by

7. REST API endpoints (prefix: /api/v1/):

   Auth:
   - POST /auth/register
   - POST /auth/login → JWT token
   - POST /auth/refresh

   Scenes:
   - GET  /scenes/{slug}           → scene + scene_objects + knowledge_nodes + asset URLs
   - GET  /scenes/{slug}/objects   → scene objects with transforms and metadata

   Knowledge:
   - GET    /knowledge-nodes/             → list/search (by slug, tag, wikidata_id)
   - GET    /knowledge-nodes/{id}         → single node
   - POST   /knowledge-nodes/             → create (auth required)
   - PUT    /knowledge-nodes/{id}         → update (auth required)
   - DELETE /knowledge-nodes/{id}         → delete (admin only)
   - GET    /knowledge-nodes/{id}/relations → related nodes

   Assets:
   - POST /assets/upload                  → chunked GLB upload (auth required)
   - GET  /assets/{id}                    → asset metadata + download URL

   Wikipedia/Wikidata Proxy:
   - GET  /wiki-proxy/search?q=...        → search Wikipedia, cache in Redis (24h TTL)
   - GET  /wiki-proxy/entity/{wikidata_id} → Wikidata entity properties, cached

PHASE 3: 3D ENGINE CORE (FRONTEND)
-----------------------------------------------------------

8. SceneViewer component:
   - Fetches scene data from GET /scenes/{slug}
   - Renders Three.js scene graph via R3F
   - Loads GLB assets via drei's useGLTF with React Suspense
   - Places scene_objects at stored transforms
   - Applies environment (HDRI, lighting, fog) from scene config

9. Camera & zoom system (drei CameraControls):
   - Orbit controls with per-scene constraints
   - Click interactive object → smooth camera animation (fitToBox/setLookAt)
   - For zoom_into objects: camera dolly + crossfade → load child scene
   - Zustand state: { currentSceneSlug, zoomStack: [...], focusedObjectId }
   - Breadcrumb/back navigation to zoom out

10. Object interaction system:
    - Hover: highlight mesh (emissive glow or outline), show floating label
      (drei <Html> component)
    - Click: expand 2D info panel showing knowledge node title, summary,
      properties from document
    - Double-click / "Go deeper" button: trigger zoom transition to child scene

11. LOD system:
    - drei <Detailed> component per scene_object
    - Swap meshes by camera distance
    - LOD variants generated at asset upload time (server-side)

PHASE 4: MVP SCENE — SCIENCE LAB
-----------------------------------------------------------

12. Scene hierarchy:

    Scene 0 — "Science Lab" (depth 0)
    └── Room with: microscope, periodic table poster, beakers, books, globe, PC

    Scene 1 — "Microscope View" (depth 1, parent: lab)
    └── Looking through lens: cell, bacteria, water molecules

    Scene 2 — "Animal Cell" (depth 2, parent: microscope)
    └── Nucleus, mitochondria, ribosomes, cell membrane

    Scene 3 — "Nucleus Interior" (depth 3, parent: cell)
    └── DNA double helix, chromosomes

    Scene 4 — "DNA Molecule" (depth 4, parent: nucleus)
    └── Procedural base pairs, individual atoms

13. 3D asset sourcing:
    - Lab room + furniture → Sketchfab CC-licensed GLB or AI-generated (Meshy/Tripo)
    - Molecules/atoms → procedural R3F geometry (instanced spheres + cylinders,
      parse PDB data from RCSB Protein Data Bank)
    - Cell/organelles → AI-generated or free biology 3D models
    - All assets processed: validate → meshopt compress → KTX2 textures → S3

14. Seed database:
    - ~20-30 knowledge nodes for the lab scene chain
      (microscope, cell, mitochondria, DNA, hydrogen atom, etc.)
    - Pull structured data from Wikidata to populate properties dict

-----------------------------------------------------------
KEY DECISIONS
-----------------------------------------------------------

- Uvicorn as the ASGI server for FastAPI (uvicorn[standard] with uvloop)
- meshopt over Draco for GLB compression (20KB decoder vs 300KB, faster)
- MongoDB as the database — natural fit for flexible/nested documents
  (scenes embed their objects, knowledge nodes embed relations)
  Use $graphLookup for traversals; Beanie ODM for async Python access
- Scene-based zoom (discrete scenes per depth level, not continuous)
- zustand over Redux (lighter, R3F-idiomatic)
- No LLM integration for now — content is manual + Wikipedia/Wikidata
- Docker Compose for dev; cloud deployment deferred until MVP validated
